# TB-002 – Build Swift Tensor Engine MVP

## Objective
Implement the core Swift tensor runtime that handles memory, math primitives, and CPU fallbacks for transformer inference. This layer should be pedagogical yet performant enough to validate kernels before Metal acceleration.

## Background
Section 4 of the PRD lists the “Swift Tensor Engine” as P0. It must expose tensor shapes, dtypes, broadcasting, and differentiable building blocks used by attention/MLP layers. Having a solid CPU-backed engine simplifies debugging and unit testing before GPU routing (Task TB-003).

## Work Items
1. Define a `Tensor` struct (or value type) with stride-aware storage, reference-counted buffers, and support for `Float16`, `Float32`, and quantized INT8 payloads.
2. Implement core ops: matmul (CPU fallback via Accelerate), elementwise add/mul, GELU, LayerNorm, Softmax, and top-k helpers, ensuring vectorized paths where possible.
3. Provide tensor initialization utilities (eye, zeros, random normal) and mmap-backed loading hooks for future weight streaming.
4. Design a modular op registry so higher layers can select CPU vs GPU implementations transparently.
5. Add comprehensive unit tests that cover shape inference, dtype conversion, broadcasting, and numerical parity (within tolerance) against reference outputs.
6. Document the architecture in `docs/overview.md` (diagram + short explanation) and include DocC articles explaining each op for educational value.

## Acceptance Criteria
- `TinyBrainRuntime` target exposes public APIs for tensor creation, inspection, and the specified ops with doc comments.
- Unit tests in `Tests/TinyBrainRuntimeTests` validate numeric accuracy (e.g., matmul relative error < 1e-4 for FP16/FP32 comparisons).
- Benchmark script (simple) reports baseline tokens/sec for CPU-only inference on a toy model and is checked into `/Scripts`.
- Documentation updates describe how tensors flow through the runtime and link to DocC articles.
- No Metal-specific dependencies are introduced yet; code compiles on simulator targets.

## Dependencies
- TB-001 completed (project scaffolding, directories, tooling).

## Notes & References
- Consult Accelerate/BNNS APIs for optimized BLAS fallback.
- Keep APIs Swift-friendly (value semantics) as highlighted in the PRD’s educational goal.
