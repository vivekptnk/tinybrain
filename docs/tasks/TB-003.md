# TB-003 – Implement Metal Kernel Suite

## Objective
Deliver GPU-accelerated kernels for the highest-impact transformer ops (MatMul, Softmax, LayerNorm, Quant/Dequant) and integrate them into the runtime’s backend abstraction so workloads automatically route to Metal on supported devices.

## Background
TinyBrain’s differentiation is a Swift-native stack with performant Metal kernels (PRD Sections 4 & 7). This task operationalizes that claim by pairing Metal shader code with Swift bindings, ensuring kernels interoperate with the tensor engine built in TB-002.

## Work Items

### Core Implementation (TB-003 Scope - Focused on MatMul)
1. Implement **tiled MatMul kernel** in `.metal` with 16×16 threadgroups using threadgroup memory optimization for educational transparency.
2. Enhance `MetalBackend.swift` to support:
   - Shader library loading and compilation
   - Buffer management (create, reuse, cleanup)
   - Command encoder pipeline for compute operations
   - Error handling and device capability checks
3. Build backend abstraction layer allowing transparent CPU/GPU operation selection:
   - Extend `Tensor.matmul()` to automatically use Metal when available
   - Fallback to CPU (Accelerate) when Metal unavailable
   - Logging hooks to debug which backend is used
4. **[Deferred from TB-002]** Implement stride-aware tensor operations (transpose, reshape) to support non-contiguous memory layouts.
5. Add comprehensive TDD tests:
   - Metal vs CPU numerical parity (< 1e-3 accuracy for Float32)
   - Performance benchmarks (Metal should be 3-10× faster than CPU)
   - Edge cases (different matrix sizes, device availability)
6. Document Metal kernel architecture, tiling strategy, threadgroup sizing in `docs/overview.md` + DocC articles.

### Deferred to Later Tasks
- **GPU Softmax kernel** → **TB-004 or later** (CPU softmax is fast enough; focus on matmul first)
- **GPU LayerNorm kernel** → **TB-004 or later** (CPU implementation sufficient for now)
- **GELU kernel** → **TB-004 or later** (minimal benefit on GPU vs CPU loop)
- **Device-specific autotuning** → **TB-007** (JSON configs, per-device tile sizes)
- **INT8 dequantization kernel** → **TB-004** (quantization task)
- **KV-cache update kernels** → **TB-004** (paged cache task)
- **Fused operations** (bias+activation, etc.) → **TB-005 or later** (optimization phase)

## Acceptance Criteria
- **MatMul Metal kernel** compiles and runs on macOS 14 and iOS 17 simulators/devices (Softmax/LayerNorm/Quant deferred).
- Runtime automatically selects Metal for matmul when available; API consumers do not reference Metal types directly (transparent backend switching).
- Benchmarks show **≥3× speedup over CPU baseline** for 512×512 and 2048×2048 MatMul on Mac M3.
- Unit/integration tests validate Metal vs CPU **numerical parity** (< 1e-3 relative error for Float32).
- Tiled implementation uses **16×16 threadgroups** with threadgroup memory optimization.
- Documentation explains Metal kernel architecture, tiling strategy, and threadgroup sizing with educational diagrams.
- Stride-aware operations (transpose/reshape) implemented to support efficient Metal memory layouts.
