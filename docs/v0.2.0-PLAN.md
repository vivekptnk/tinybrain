# TinyBrain v0.2.0 — Development Plan

## Goals

Make TinyBrain work with more models, run faster, and use less memory.

---

## Phase 1: INT4 Quantization

**Why:** 8x memory savings vs FP32 (vs 4x for INT8). Enables running larger models on iPhones.

**What:**
- Per-group INT4 quantization (group size 128)
- FP16 scales per group, INT4 zero points
- Metal kernel for INT4 dequant + matmul (fused)
- Converter script support (`--quantize int4`)

**Acceptance:**
- < 5% perplexity increase vs FP16
- TinyLlama fits in ~550MB (vs 1.1GB INT8)
- All existing tests still pass

**Estimated effort:** 1-2 weeks

---

## Phase 2: More Models

**Why:** People want to use their own models, not just TinyLlama.

**What:**
- Validate and document: Gemma 2B, Phi-2 2.7B, Mistral 7B (INT4)
- SentencePiece tokenizer adapter (for Google models)
- TikToken adapter (for OpenAI-style models)
- Model picker UI in the demo app

**Acceptance:**
- At least 3 models working end-to-end
- Tokenizer auto-detection handles all formats
- Model picker shows available .tbf files in Models/

**Estimated effort:** 2-3 weeks

---

## Phase 3: Performance

**Why:** Faster inference = better user experience, especially on phones.

**What:**
- FlashAttention Metal kernel (fused Q×K^T + softmax + ×V)
- Metal kernels for softmax, layernorm, GELU (currently CPU)
- Speculative decoding (generate multiple candidates, verify in parallel)
- Device-specific tuning (threadgroup sizes per Apple Silicon generation)

**Acceptance:**
- 2x+ throughput improvement on M1
- All kernels have CPU fallbacks
- No accuracy regression

**Estimated effort:** 3-4 weeks

---

## Phase 4: Core ML Hybrid

**Why:** Apple's ANE (Neural Engine) can accelerate attention on newer chips.

**What:**
- Core ML model for attention computation
- Dynamic dispatch: Metal for matmul, ANE for attention
- Benchmark to verify ANE actually helps (it doesn't always)
- Fallback to pure Metal when Core ML unavailable

**Acceptance:**
- Measurable speedup on at least one device class
- No regression on devices without ANE
- Clean abstraction (user doesn't need to know which backend runs)

**Estimated effort:** 2-3 weeks

---

## Phase 5: Polish

**What:**
- Energy benchmarking (J/token via MetricsKit)
- iOS app target (not just macOS)
- TestFlight build
- Swift Playgrounds integration for education
- Screenshot/GIF for README

**Estimated effort:** 1-2 weeks

---

## Priority Order

1. INT4 Quantization (biggest user impact)
2. More Models (biggest adoption driver)
3. Performance (biggest technical impact)
4. Core ML Hybrid (nice to have)
5. Polish (ongoing)

## Timeline

Rough estimate: 10-14 weeks for the full v0.2.0 scope. Phases can ship independently as minor releases (0.2.0, 0.3.0, etc.).
